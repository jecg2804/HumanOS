# Notifications architecture: in-app + email PRIMARY MVP, Vercel Cron worker, SMS/WhatsApp diferidos v1.1

**Revised 2026-05-27 post-audit**: email worker pasa de Supabase Edge Function a Vercel Cron + Next.js route handler (single-source templates, alineado con stack del app). Domain corregido a `rein-eisenwerk.com` (verificado en Resend). `RESEND_FROM_EMAIL` reemplaza `RESEND_FROM_ADDRESS`. Reply-To pattern añadido.

NotificationEngine inserta filas en `notifications.outbox` con `channel IN ('in_app', 'email')` como canales primarios MVP. SMS y WhatsApp existen en el CHECK del schema pero quedan `status='pending'` indefinido — ningún worker MVP los procesa. Patrón fire-and-forget (R18): la server action que dispara la notification ejecuta el INSERT en la misma transacción que la operación principal, pero la entrega externa (email vía Resend, render in-app via Realtime) ocurre asíncrona. Si Resend o Realtime fallan, la operación de negocio (crear ticket, actualizar status, completar onboarding) SUCCESS de todas formas — la fila `outbox` queda `status='failed'` con `error_message` para retry humano.

**in_app**: cero worker. Frontend se suscribe via Supabase Realtime a `notifications.outbox WHERE recipient_id = current_person_id() AND channel='in_app'`. RLS de `outbox` filtra a recipient + hr_admin. `NotificationBell` component en `AppShell` muestra unread count; `markAsRead()` server action `UPDATE status='read', read_at=now()`. Sin polling, sin colas externas.

**email**: worker via **Vercel Cron + Next.js route handler** (`/api/cron/process-notifications`). Schedule MVP: cada 5 minutos vía cron expression `*/5 * * * *` declarado en `vercel.ts`. Razón del intervalo: tráfico MVP bajo (6 bootstrap codes + onboarding gradual), cada-60-segundos sería overkill en invocations cost. Cuando el volumen crezca post-MVP el intervalo se afina sin tocar arquitectura. Cada tick procesa rows `WHERE channel='email' AND status='pending' AND attempts < max_attempts`. Llama `resend.emails.send({ from, to, reply_to, subject, react: <Template /> })` pasando el componente React directamente (Resend v6 SDK soporta `react` key, evita render manual). Retry exponential: attempts++ por intento, marca `status='failed'` permanente al alcanzar `max_attempts` (default 3). Templates viven en `src/emails/*.tsx` usando `@react-email/components` (Html, Body, Container, Heading, Text, Button, Tailwind con `pixelBasedPreset`). `template_code` column en outbox maps al filename (`onboarding_error_reported` → `src/emails/OnboardingErrorReported.tsx`). `template_variables` JSONB son las props. Templates importados directamente desde `src/emails/` (single source of truth) — el route handler corre en Node.js runtime con acceso al filesystem de Next.js, sin necesidad de sync scripts.

**Auth del cron**: route handler valida `x-vercel-cron` header presente Y `Authorization: Bearer ${process.env.CRON_SECRET}`. Vercel firma cron invocations automáticamente; el CRON_SECRET previene invocations externas manuales por curl.

**Per-user opt-in**: reusamos `hr.user_settings.preferences` JSONB existente con namespace `notifications`, shape `{ notifications: { email: { <notification_type>: bool }, sms: {}, whatsapp: {} } }`. `in_app` no es opt-out-able (siempre on, sistema operacional). NotificationEngine antes de insertar email channel consulta preferences; si type está off para el recipient, omite el INSERT email (sí inserta in_app). UI F33 (Group 3) renderiza los toggles email y muestra "Próximamente" para SMS/WhatsApp. **No se crea columna nueva**: los 4 booleanos legacy `notification_{email,in_app,sms,whatsapp}_enabled` quedan como "channel master switch" (cuando F33 los cablee). Per-type granularity vive en `preferences->'notifications'`.

**Pattern INSERT**: cada server action que muta business state (`onboardEmployee`, `createTicket`, `approveTicket`, `regenerateInviteCode`, etc.) inserta sus notifications correspondientes en `notifications.outbox` dentro de la MISMA transacción que la mutación principal. Si la mutación hace ROLLBACK, las notifications también ROLLBACK — no se entrega aviso de un evento que no sucedió. NO se usan triggers BD para INSERT en outbox: la server action retiene contexto del caller (`actor_id`, request id, source operation semantic, idempotency keys cuando aplique) que triggers no preservan limpiamente. Esto también facilita testing — un test del flow business observa simultáneamente la mutación + las filas en outbox sin necesidad de mockear capa BD.

**Reply-To pattern**: todos los emails llevan `reply_to: process.env.RESEND_REPLY_TO` (`samantha.kosmas@iconsanet.com`) **excepto** los de `password_reset` (self-service, no se contesta). El worker aplica:

```typescript
const replyTo = row.template_code === 'password_reset' ? undefined : process.env.RESEND_REPLY_TO;
await resend.emails.send({ from, to, ...(replyTo ? { reply_to: replyTo } : {}), subject, react: <Template ... /> });
```

**Idempotencia + retry-friendly**: cuando la server action que dispara una notification ya commited el outbox INSERT y luego algo upstream falla (caso: server action multi-paso donde paso N+1 falla post-outbox-insert), el rollback de transacción descarta el outbox row también — así no hay duplicados. Para multi-table writes que requieren atomicity adicional (caso `onboardEmployee` con auth.admin + RPC), la solución del rollback gap vive en ADR-0006 (capture-then-restore + RPC idempotente), no en este ADR.

Alternativa rechazada (a): defer notifications a Group 4 engines, usar email manual (workflow analógico) para los primeros 6 bootstrap codes. Funciona para piloto pero no escala a los 184 empleados pre-MVP que se onboardearán post-MVP. Tener email wired desde Group 2 evita un refactor disruptivo cuando se quiera "activar notifications".

Alternativa rechazada (b — revisited): Supabase Edge Function. Trade-offs descubiertos durante audit Group 2: (1) MovimientOS no usa Edge Functions deployed (Supabase project tiene cero functions activas hasta este punto), así introduciríamos un nuevo runtime sin precedente; (2) templates React Email en `src/emails/` requieren sync script para que la Edge Function (Deno) los compile — dual source riesgo de drift; (3) Vercel Cron + Next.js route corre en el mismo runtime Node que server actions y comparte env vars natively; (4) `vercel.ts` (config nuevo per knowledge update 2026) declara crons declarativamente. La preferencia inicial por Edge Function se basaba en "corre cerca de BD" pero la latencia adicional de cross-region HTTP del cron es trivial vs el costo operacional del dual-source. Decisión final: Vercel Cron.

Alternativa rechazada (c): scheduled Postgres pg_cron + LISTEN/NOTIFY. Demasiado bajo nivel para templates React. Reservado para tareas BD-internas sin UI (ej. limpieza de tokens expirados, agregación de métricas).

Riesgo conocido: si Resend domain verification para `rein-eisenwerk.com` no está completo al deploy, emails fallan silenciosamente con status='failed'. Mitigación: route handler pre-flight check al arrancar (`if (!process.env.RESEND_API_KEY || !process.env.RESEND_FROM_EMAIL) { log.warn(...); return; }`), y verificación operacional pre-cutover de que `rein-eisenwerk.com` está marcado como Verified en Resend dashboard.
