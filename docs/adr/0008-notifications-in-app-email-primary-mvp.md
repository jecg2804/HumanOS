# Notifications architecture: in-app + email PRIMARY MVP, SMS/WhatsApp diferidos v1.1

NotificationEngine inserta filas en `notifications.outbox` con `channel IN ('in_app', 'email')` como canales primarios MVP. SMS y WhatsApp existen en el CHECK del schema pero quedan `status='pending'` indefinido — ningún worker MVP los procesa. Patrón fire-and-forget (R18): la server action que dispara la notification ejecuta el INSERT en la misma transacción que la operación principal, pero la entrega externa (email vía Resend, render in-app via Realtime) ocurre asíncrona. Si Resend o Realtime fallan, la operación de negocio (crear ticket, actualizar status, completar onboarding) SUCCESS de todas formas — la fila `outbox` queda `status='failed'` con `error_message` para retry humano.

**in_app**: cero worker. Frontend se suscribe via Supabase Realtime a `notifications.outbox WHERE recipient_id = current_person_id() AND channel='in_app'`. RLS de `outbox` filtra a recipient + hr_admin. `NotificationBell` component en `AppShell` muestra unread count; `markAsRead()` server action `UPDATE status='read', read_at=now()`. Sin polling, sin colas externas.

**email**: worker via Supabase Edge Function scheduled (preferido sobre Vercel Cron porque ya tenemos `pg_cron` en plan Pro + Edge Functions corren cerca de la BD). Schedule MVP: cada 5 minutos vía cron expression `*/5 * * * *`. Razón del intervalo: tráfico MVP bajo (6 bootstrap codes + onboarding gradual), cada-60-segundos sería overkill en invocations cost. Cuando el volumen crezca post-MVP el intervalo se afina sin tocar arquitectura. Cada tick procesa rows `WHERE channel='email' AND status='pending' AND attempts < max_attempts`. Llama `resend.emails.send({ from, to, subject, react: <Template /> })` pasando el componente React directamente (Resend v6 SDK soporta `react` key, evita render manual). Retry exponential: attempts++ por intento, marca `status='failed'` permanente al alcanzar `max_attempts` (default 3). Templates viven en `src/emails/*.tsx` usando `@react-email/components` (Html, Body, Container, Heading, Text, Button, Tailwind con `pixelBasedPreset`). `template_code` column en outbox maps al filename (`onboarding_error_reported` → `src/emails/OnboardingErrorReported.tsx`). `template_variables` JSONB son las props.

**Per-user opt-in**: `hr.user_settings.notification_preferences` JSONB con shape `{ email: { <notification_type>: bool }, sms: {}, whatsapp: {} }`. `in_app` no es opt-out-able (siempre on, sistema operacional). NotificationEngine antes de insertar email channel consulta preferences; si type está off para el recipient, omite el INSERT email (sí inserta in_app). UI F33 (Group 3) renderiza los toggles email y muestra "Próximamente" para SMS/WhatsApp.

**Pattern INSERT**: cada server action que muta business state (`onboardEmployee`, `createTicket`, `approveTicket`, `regenerateInviteCode`, etc.) inserta sus notifications correspondientes en `notifications.outbox` dentro de la MISMA transacción que la mutación principal. Si la mutación hace ROLLBACK, las notifications también ROLLBACK — no se entrega aviso de un evento que no sucedió. NO se usan triggers BD para INSERT en outbox: la server action retiene contexto del caller (`actor_id`, request id, source operation semantic, idempotency keys cuando aplique) que triggers no preservan limpiamente. Esto también facilita testing — un test del flow business observa simultáneamente la mutación + las filas en outbox sin necesidad de mockear capa BD.

Alternativa rechazada (a): defer notifications a Group 4 engines, usar email manual (workflow analógico) para los primeros 6 bootstrap codes. Funciona para piloto pero no escala a los 184 empleados pre-MVP que se onboardearán post-MVP. Tener email wired desde Group 2 evita un refactor disruptivo cuando se quiera "activar notifications".

Alternativa rechazada (b): Vercel Cron Jobs en lugar de Supabase Edge Functions. Vercel Cron es por route handler HTTP con auth via header — funciona pero (1) agrega latencia (HTTP cross-region vs Edge Function pg-local), (2) cuenta contra cuota Vercel Functions, (3) duplica capability que ya tenemos en plan Supabase Pro. Edge Function programada con `supabase functions deploy ... --schedule '* * * * *'` es más directo.

Riesgo conocido: si Resend domain verification para `iconsa.com.pa` no está completo al deploy, emails fallan silenciosamente con status='failed'. Mitigación: Edge Function pre-flight check al arrancar (`if (!process.env.RESEND_API_KEY) { log.warn(...); return; }`), y un J-item operacional explícito en `docs/03-ROADMAP-POST-MVP.md` confirmando que `iconsa.com.pa` está verificado en Resend dashboard antes de cutover.

Decisión `from` address pending James + Samantha durante Resend domain verification (candidates: `notificaciones@iconsa.com.pa`, `humanos@iconsa.com.pa`, `no-reply@iconsa.com.pa`). Default placeholder en código: `process.env.RESEND_FROM_ADDRESS`. Edge Function valida presencia de esta env var en pre-flight check; si está missing, log warning y skip ciclo (no error que rompa el cron). Sin from address válido, in-app sigue funcionando — degradación graceful.
