# employment_type como tabla referencia con metadata operacional, no CHECK enum

`hr.employment_types` es una tabla referencia con 4 rows (`tiempo_indefinido`, `tiempo_definido`, `obra_determinada`, `servicios_profesionales`) cada una con columnas booleanas + porcentajes que codifican reglas operacionales del SOP IC-RH-D-05: `has_vacations`, `has_xiii_month`, `has_seniority_premium`, `ss_applies`, `se_applies`, `isr_applies`, `severance_pct`, `union_fee_pct`. `hr.employments.employment_type_id` es FK estricto contra esta tabla (ON DELETE RESTRICT). La elección es deliberada vs el patrón usado en otras columnas similares (`app_role` con CHECK `IN (...)`).

Razón: F-05-01 (Acción Personal), F-05-02 (Préstamo), F-05-03 (Vacaciones) tienen lógica condicional por tipo de contrato (`servicios_profesionales` no aplica vacaciones, `obra_determinada` paga 6% indemnización, etc.). Codificar esta lógica en `if (employmentType === 'servicios_profesionales')` regado por toda la app es frágil: cuando Samantha cambie una regla operacional (ej. CCT 2027 ajusta % indemnización), hay que cazar todos los call sites. Con tabla referencia + metadata, FormEngine + ApprovalEngine consultan `hr.employment_types` UNA VEZ al cargar el ticket y aplican la regla derivada del row. Cambio de policy = single UPDATE en BD, sin migration de código.

Alternativa rechazada: CHECK constraint enum-like (`employment_type TEXT CHECK (employment_type IN ('tiempo_indefinido', 'tiempo_definido', 'obra_determinada', 'servicios_profesionales'))`). Más simple pero pierde la capacidad de portar metadata. La lógica operacional quedaría hardcoded en TS/SQL functions, violando single-source-of-truth.

Trade-off: agregar un nuevo tipo requiere `INSERT INTO hr.employment_types` (DDL light, no migration de schema) más actualizar FormEngine si hay UI especial. Dominio es cerrado por SOP así que añadir tipos es raro — beneficio supera al costo. `*_text` fallback (como `position_text` para `position_id`) NO aplica: `employment_type_id` es estricto sin fallback, porque la metadata operacional la necesita downstream y un fallback string lo rompería.
